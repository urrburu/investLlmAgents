# 출력 제약 및 MVP 경계

## 출력 제약: 추천이 아닌 현황 점검

### 허용되는 출력

- 현재 포트폴리오 구성과 변화
- 종목별 최근 이벤트 요약
- 책 원칙과 현재 행동의 충돌 가능성
- 과거 매매 실수와 유사한 패턴
- 확인해야 할 질문 목록
- 계산 기준일, 사용한 출처, 누락된 데이터
- 검증 상태와 사람이 확인해야 하는 이유

### 금지되는 출력

- 지금 매수/매도하라는 직접 지시
- 목표가 / 수익률 보장
- 자동매매 실행
- 근거 없는 확정적 전망
- 출처 없는 투자 판단
- 검증 실패를 숨긴 매끄러운 최종 답변
- 사용자의 투자 성향이나 법적 지위를 단정하는 표현

## MVP 경계

| 단계 | 포함 | 제외 |
|---|---|---|
| **MVP 1** | 책·매매일지·포트폴리오 업로드, 청킹/임베딩, 위키 초안, 포트폴리오 현황 리포트 | 자동 매매, 목표가 제시, 완전 자동 발행 |
| **MVP 2** | 정기 실행, 기존 리포트와 비교, 종목별 Snapshot, Persona Formatter | 복잡한 파생상품 분석, 세금 최적화 |
| **MVP 3** | GitHub/Notion/브로커 API 연동, 고도화된 검증, 대시보드 | 규제상 투자자문으로 해석될 수 있는 자동 의사결정 |

## MVP 완료 기준

| 단계 | 완료 기준 |
|---|---|
| **MVP 1** | 1개 이상 원문을 위키 초안으로 바꾸고, 포트폴리오 리포트가 숫자/출처 검증을 통과한다. |
| **MVP 2** | 정기 실행 결과가 이전 리포트와 비교되고, 종목 Snapshot이 thesis와 리스크를 연결한다. |
| **MVP 3** | 외부 연동 실패가 사용자에게 명확히 보이고, 대시보드에서 검증 상태와 누락 데이터를 확인할 수 있다. |

## 품질 게이트

- 모든 숫자에는 계산 함수와 원천 데이터 timestamp가 있어야 한다.
- 모든 주요 주장에는 `source_refs`가 있어야 한다.
- `verification_status`가 `passed`가 아니면 최종 리포트가 아니라 검토 요청으로 출력한다.
- 최신성이 중요한 데이터는 `as_of`를 표시한다.
- 금지 표현 검사를 통과하지 못한 문장은 Persona Formatter로 보내지 않는다.

## 구현 우선순위

1. **데이터 모델** — 책, 매매일지, 포트폴리오, 위키 페이지 스키마
2. **RAG 파이프라인** — 청킹, 임베딩, 검색, citation 연결
3. **위키 갱신 Agent** — 원칙/패턴 추출과 페이지 revision 생성
4. **포트폴리오 점검 Agent** — 계산 + 위키 참조 + 리포트 생성
5. **검증 / 스타일** — 숫자/출처 검증 후 Persona 포맷팅

## 명시적으로 뒤로 미루는 것

- 브로커 주문 API와 자동매매
- 목표가/매수/매도 추천 UX
- 세금 최적화와 파생상품 분석
- 여러 사용자 계정과 권한 모델
- 법률/세무/투자자문 문서 자동 생성
