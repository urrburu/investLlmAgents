# Agents 스펙

## Agent 목록

| Agent | Trigger | 읽는 지식 | 호출하는 Skills | 출력 |
|---|---|---|---|---|
| **LLMwiki 갱신 Agent** | 책·일지·문서 업로드 | 기존 위키, 새 원문 | 파싱, 청킹, 임베딩, entity 추출, citation 검증 | 위키 페이지 생성/수정 초안 |
| **정기 점검 Agent** | 매일/매주 스케줄 | 투자 원칙, 매매 패턴, 현재 포트폴리오 | 가격 조회, 포트폴리오 계산, RAG 검색, 리포트 생성 | 현황 점검 리포트 |
| **시황 분석 Agent** | 매일 장전/장후 또는 사용자 요청 | 시장 위키, 과거 시황 메모, 포트폴리오 노출, 투자 원칙 | 시장 데이터 조회, 매크로 지표 수집, Risk-on/Risk-off 판단, 시황 요약 | 시장 국면 판단과 포트폴리오 점검 포인트 |
| **종목 분석 Agent** | 사용자 요청 | 종목 위키, thesis, 관련 원칙, 과거 메모 | 티커 정규화, 뉴스/공시 수집, 재무/가격 조회, 출처 검증 | 종목 Snapshot / 변화 요약 |
| **포트폴리오 분석 Agent** | 사용자 요청 또는 정기 실행 | 포트폴리오 위키, 매매일지, 투자 규칙 | 비중 계산, 수익률 계산, 집중도 분석, 리스크 감지 | 포트폴리오 상태와 점검 항목 |
| **검증 Agent** | 모든 리포트 생성 후 | 계산 결과, 원문 chunk, citation | 숫자 검증, citation 검증, 금지 표현 검사 | 검증 통과/보류/수정 요청 |
| **Persona Formatter Agent** | 최종 응답 직전 | 검증된 분석 결과 | 스타일 변환, 길이 조정, 출력 포맷팅 | 스타일화된 답변 |

## Agent 공통 실행 계약

모든 Agent 실행은 같은 외곽 계약을 따른다.

| 항목 | 계약 |
|---|---|
| 입력 | `run_id`, `trigger_type`, 사용자 요청 또는 스케줄 정보, 접근 가능한 데이터 범위 |
| 상태 | Graph state에 중간 결과, 오류, 검증 상태를 명시적으로 저장 |
| 출력 | 최종 답변 또는 revision 제안과 함께 `verification_status`를 반환 |
| 금지 | 검증 전 최종 출력, 출처 없는 주장, 자동 매매 실행 |
| 관측성 | 시작, 외부 데이터 조회, 검증 실패, 최종 상태를 이벤트로 남김 |

## Agent별 실패 처리

| Agent | 대표 실패 | 처리 |
|---|---|---|
| LLMwiki 갱신 Agent | 원문 파싱 실패, citation 부족 | 해당 문서를 `needs_human_review`로 남기고 위키 저장을 보류 |
| 정기 점검 Agent | 가격 데이터 누락, 포트폴리오 파일 오류 | 누락 항목을 분리 표시하고 계산 가능한 범위만 부분 리포트 생성 |
| 시황 분석 Agent | 시장 데이터 일부 지연, 지표 충돌 | 국면을 확정하지 않고 `mixed` 또는 `insufficient_data`로 표시 |
| 종목 분석 Agent | ticker 모호성, 뉴스 출처 불충분 | 사용자가 확인할 후보 ticker와 누락 출처를 반환 |
| 포트폴리오 분석 Agent | 보유 수량/평단 누락 | 수익률 계산을 보류하고 비중/노출처럼 가능한 계산만 수행 |
| 검증 Agent | 숫자 불일치, 근거 없는 주장 | 최종 출력을 차단하고 수정 요청 목록을 생성 |
| Persona Formatter Agent | 금지 표현 발생 | 표현을 수정하거나 검증 Agent로 되돌림 |

## Agent가 직접 판단하지 않는 것

- 매수/매도 실행 여부
- 목표가와 보장 수익률
- 사용자의 투자 성향을 법적 의미의 자문 등급으로 분류하는 일
- 출처가 없는 시장 전망을 확정 표현으로 바꾸는 일

## 계층 내 역할 구분

| 구분 | Agent |
|---|---|
| 정체 | 작업 흐름을 실행하는 orchestrator |
| 예시 | "오늘 포트폴리오 점검" 실행 |
| 상태 | 실행 중 상태를 가짐 |
| 변경 주체 | Scheduler 또는 사용자 요청으로 실행 |
| 성공 기준 | 업무 흐름 완료 및 검증 통과 |
