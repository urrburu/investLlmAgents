# 시스템 개요 (Overview)

## 한 문장 정의

> **LLMwiki**는 "나의 투자 지식 저장소"이고, **Agents**는 "그 지식을 사용해 판단 흐름을 실행하는 작업자"이며, **Skills**는 "Agent가 호출하는 개별 기능"이다.

## 세 계층 요약

| 계층 | 역할 | 상태 |
|---|---|---|
| **LLMwiki** (Knowledge Layer) | 투자 원칙·매매일지·포트폴리오·종목 thesis 저장 | 장기 저장 |
| **Agents** (Workflow Layer) | LangGraph로 정기 점검·종목 분석·포트폴리오 분석·위키 갱신 실행 | 실행 중 상태 보유 |
| **Skills** (Tool Layer) | Agent가 호출하는 단위 기능 (계산·검색·청킹·임베딩·검증·포맷팅) | 보통 stateless |

## 핵심 원칙

- 숫자는 **코드**가 계산
- 근거는 **LLMwiki / RAG**에서 검색
- LLM은 **해석과 요약** 담당
- LangGraph는 **흐름과 검증** 담당
- PostgreSQL은 **canonical 데이터 저장소** 담당
- 시황분석은 **Risk-on/Risk-off 판단** 후 포트폴리오 노출과 연결

## CEO 리뷰 관점의 제품 판단

이 시스템은 "투자 추천 엔진"이 아니라 **개인 투자 메모리와 점검 루틴을 실행하는 감사 가능한 작업대**로 정의한다. 사용자가 얻어야 하는 1차 가치는 매수/매도 신호가 아니라, 내가 이미 정한 원칙과 현재 행동이 어디서 어긋나는지 빠르게 보는 것이다.

| 판단 | 결정 |
|---|---|
| 핵심 사용자 가치 | 투자 원칙, 매매일지, 현재 포트폴리오를 연결해 자기 점검 시간을 줄인다. |
| 차별점 | 일반 시황 요약이 아니라 "내 원칙/내 실수/내 보유 노출"과 연결한다. |
| 제품 경계 | 직접적인 매매 지시, 목표가, 수익 보장, 자동 주문은 범위 밖이다. |
| MVP 성공 기준 | 출처가 연결된 위키 초안과 검증된 포트폴리오 점검 리포트를 안정적으로 만든다. |
| 실패 기준 | 근거 없는 전망, 계산 불일치, 출처 없는 주장을 최종 출력으로 내보낸다. |

## 10x 방향과 MVP 제약

장기적으로는 사용자의 투자 원칙이 살아 있는 지식 그래프처럼 작동해야 한다. 좋은 상태는 Agent가 "오늘은 A 종목을 사라"가 아니라 "이 판단은 과거 손실 패턴 B와 닮았고, 네가 적어둔 원칙 C와 충돌한다"를 근거와 함께 보여주는 것이다.

MVP에서는 이 비전을 좁게 구현한다. 먼저 데이터 모델, RAG, 위키 revision, 포트폴리오 점검, 검증 게이트를 완성하고, 외부 브로커 연동이나 자동 의사결정은 뒤로 미룬다.

MVP의 저장소는 PostgreSQL로 고정한다. 위키, revision, 리포트, 검증 결과, Agent 실행 상태는 PostgreSQL row로 남기고, Markdown은 export 또는 사용자에게 보여주는 출력 형식으로만 사용한다.

## 입력 소스

- 투자서 PDF / 메모
- 매매일지
- 현재 포트폴리오
- 과거 분석 리포트
- 뉴스 / 공시 / 가격 데이터

## 출력 형태

- 일간 / 주간 현황 점검
- 종목 Snapshot
- 포트폴리오 리스크 리포트
- 위키 페이지 초안
- 확인 필요 항목 목록

## 문서 간 구현 계약

- 데이터 구조는 [07-data-contracts.md](07-data-contracts.md)를 기준으로 한다.
- 출력 안전성과 검증 실패 처리는 [08-verification-and-safety.md](08-verification-and-safety.md)를 기준으로 한다.
- Graph는 검증 통과 전에는 최종 응답을 만들 수 없다.
- Agent는 위키를 직접 덮어쓰지 않고 revision 제안만 만든다.
